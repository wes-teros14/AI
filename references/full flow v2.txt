STEP 1: rewrite_query(query, session_id)
═══════════════════════════════════════════════════════════
Purpose: Resolve vague follow-up queries into standalone
         search queries using the LLM.
When:    Only fires when chat history exists.
         First question always passes through unchanged.
How:     Sends last 2 exchanges (4 messages max) from chat
         history + current query to LLM with a rewrite prompt.
         LLM produces a standalone query preserving all
         technical terms, proper nouns, and domain-specific
         names (e.g. ExternalTimeData, correctionScenario).
Cost:    ~0.3-0.5s extra per query on Groq free tier.
         Skipped entirely on first question (no history).

Example:
  History: "What does correctionScenario do in ExternalTimeData?"
  Query:   "What happens if I don't set it during POST?"
  Output:  "What happens if I don't set the correctionScenario
            field in ExternalTimeData during a POST request?"
                           │
                           ▼


═══════════════════════════════════════════════════════════
STEP 2: expand_query(query)
═══════════════════════════════════════════════════════════
Purpose: Resolve known SAP aliases to canonical terms
         before retrieval. Ensures both BM25 and Vector
         search on full names, not shorthand.
How:     Regex word-boundary replacement (\b) with
         re.IGNORECASE flag. Processes all aliases in a
         single pass through the replacements dict.

Aliases:
  SCI, SCPI  → SAP Cloud Integration CPI
  SF, SFSF   → SuccessFactors Employee Central
  EC         → Employee Central SuccessFactors
  S4         → S/4HANA S/4
  ECP        → Employee Central Payroll
  BIB        → Business Integration Builder replication
  PTP        → Point-to-Point replication
  SCC        → SAP Cloud Connector
  "IS"       → removed — too short, matches common English

Example:
  Input:  "BIB replication data sent from CPI but S4
           is not receiving it"
  Output: "Business Integration Builder replication data
           sent from CPI but S/4HANA S/4 is not receiving it"
                           │
                           ▼


═══════════════════════════════════════════════════════════
STEP 3: Deduplication regex (inside expand_query)
═══════════════════════════════════════════════════════════
Purpose: Clean up repeated words caused by alias expansion.
         When query already contains expanded term and alias
         also expands to same word, duplicates appear.
How:     re.sub(r'\b(\w+)\s+\1\b', r'\1', ...)
         Collapses consecutive duplicate words into one.
         Case-insensitive via re.IGNORECASE.

Example:
  Before: "Business Integration Builder replication replication"
  After:  "Business Integration Builder replication"

                           ▼

STEP 4: HYBRID SEARCH (inside build_retriever)
═══════════════════════════════════════════════════════════
Two parallel search paths run on the expanded query.
Results are merged by weighted score.

┌─────────────────────────────────────────────────────┐
│                                                     │
│  BM25 (Keyword)          Vector (Semantic)          │
│  bm25_retriever          Chroma cosine similarity   │
│  .invoke(bm25_query)     .similarity_search_with_   │
│                           relevance_scores(query)   │
│  Fetches THRESHOLD_K*2   Fetches k*2 candidates     │
│  candidates for merging  for merging                │
│                                                     │
│  ┌──────────────────┐                               │
│  │ strip_stopwords() │                              │
│  │                    │                              │
│  │ Applied to BM25    │                              │
│  │ query ONLY.        │                              │
│  │ Removes common     │                              │
│  │ English filler     │                              │
│  │ words so BM25      │                              │
│  │ focuses on         │                              │
│  │ meaningful terms.  │                              │
│  │                    │                              │
│  │ Stripped words:     │                              │
│  │ do, you, have,     │                              │
│  │ any, some, the,    │                              │
│  │ a, an, is, are,    │                              │
│  │ on, in, for, of,   │                              │
│  │ to, my, your,      │                              │
│  │ notes, about,      │                              │
│  │ what, how, does,   │                              │
│  │ can, could, would, │                              │
│  │ should, it, its,   │                              │
│  │ this, that, there, │                              │
│  │ with, from, was,   │                              │
│  │ were, been, being, │                              │
│  │ has, had, did,     │                              │
│  │ tell, me, know,    │                              │
│  │ get, got, i, we    │                              │
│  │                    │                              │
│  │ Vector search gets │                              │
│  │ the FULL query —   │                              │
│  │ no stripping.      │                              │
│  │                    │                              │
│  │ Safety: if all     │                              │

│  └──────────────────┘                               │
│                                                     │
│  BM25 scoring:           Vector scoring:            │
│  Rank-based normalised   Chroma returns 0.0-1.0     │

│                                                     │
│  ── Merge ────────────────────────────────────────  │
│  Unique docs collected from both sources.           │
│  Deduplication by first 100 chars of page_content.  │
│  If same chunk found in both BM25 and Vector,       │
│  BM25 version kept (first seen wins).               │
│                                                     │
│  ── Weighted hybrid score per doc ────────────────  │
│  combined = (BM25_WEIGHT x bm25_score)              │

│                                                     │
│  All three scores attached to doc.metadata:         │

│                                                     │

│                                                     │

═══════════════════════════════════════════════════════════
STEP 7: LLM INFERENCE (build_rag_chain)
═══════════════════════════════════════════════════════════

format_docs():
  Joins all retrieved chunks with --- separators.
  Each chunk prefixed with [Source: filename].

Chain structure:
  context      ← format_docs(retrieved chunks)
  question     ← user query
  chat_history ← session memory via RunnableWithMessageHistory
      │
      ▼
  SYSTEM_PROMPT with {context} injected
      │
      ▼
  get_llm() → Groq llama-3.3-70b-versatile
               temperature=0
               request_timeout=60s
               max_retries=2
      │
      ▼
  StrOutputParser → raw string response

Session memory:
  get_session_history(session_id)
  Stores full conversation per session.
  Cleared via 'clear' command.
  Used by rewrite_query() for follow-ups.
  Used by RunnableWithMessageHistory for
  conversational continuity in LLM responses.

How LLM handles chunks:
  1 entry in chunk:
    → LLM reads entry and answers directly.

  2+ entries in chunk:
    → Happens when short related entries merge
      below CHUNK_SIZE=1500 chars.
    → Audit preview shows first 120 chars only,
      so second entry not visible in audit.
    → LLM reads ALL entries in the chunk,
      matches each heading against the question,
      selects the relevant heading,
      answers only from that entry,
      ignores all other entries in the chunk,
      reports selected heading as Source Entry.
                           │
                           ▼


═══════════════════════════════════════════════════════════
STEP 8: SYSTEM PROMPT RULES
═══════════════════════════════════════════════════════════

PHASE 1 — KNOWLEDGE RETRIEVAL:
  * Use entry headings to identify source
  * Apply alias resolution (second pass in LLM):
      SF, SFSF = Employee Central
      CPI, SCI, SCPI, IS = SAP Integration Suite
      BIB = Business Integration Builder Replication
      PTP = Point-to-Point Replication
      SCC = SAP Cloud Connector
  * Conflicts → more specific entry wins

PHASE 2 — STRICT GROUNDING:
  * No general SAP training knowledge for technical answers
  * No hallucinated TCodes, fields, entities, or steps
  * No answer → output ONLY the fallback line:
    "I do not have enough information in your
     Knowledge Base to answer this."
    No Source Entry, no Consultant Insight,
    no Suggested Next Step, no additional commentary.

PHASE 3 — RESPONSE RULES:
  * Clean Markdown, backticks for technical terms
  * No hyphens in prose, no emojis in code
  * References only from same Source Entry heading
    (do not pull from adjacent entries in chunk)

OUTPUT FORMAT:
  * Source Entry
  * Answer
  * References (omit if none)
  * Consultant Insight
  * Suggested Next Step

═══════════════════════════════════════════════════════════
STEP 9: POST-PROCESSING GUARD
═══════════════════════════════════════════════════════════
Applied after LLM response, before printing.
Defensive safety net — catches LLM non-compliance.

Check 1 — Exact fallback phrase detected:
  "I do not have enough information in your
   Knowledge Base to answer this."
  → Strip EVERYTHING else from response.
    Output ONLY the fallback line.
    Prevents LLM from adding Consultant Insight
    or Suggested Next Step after admitting no answer.

Check 2 — Hedge patterns detected:
  * "does not explicitly mention"
  * "does not contain information"
  * "no direct mention"
  * "not explicitly covered"
  → Force clean fallback response.
    Catches cases where LLM recognises the context
    is wrong but still tries to answer from it
    instead of using the exact fallback phrase.

Neither detected:
  → Response passes through unchanged.
                           │
                           ▼

FUNCTION REFERENCE
═══════════════════════════════════════════════════════════

STARTUP AND CONFIG:
  main()                 → Startup, argparse, conversation loop, post-processing guard.

LLM:
  get_llm()              → LLM factory. Returns Groq or Gemini based on SELECTED_LLM.

UI:
  print_header()         → Startup banner: status, model, memory, search weights.
  print_audit_log()      → Chunk-level retrieval diagnostics with all three scores.

DATA INGESTION:
  make_splitter()        → Shared RecursiveCharacterTextSplitter for ingest and BM25.
  ingest_data()          → Load, split, embed, store. Returns (vector_db, chunks).
  load_vectorstore()     → Load existing Chroma DB from disk.
  load_chunks_for_bm25() → Re-split docs for BM25 index (rebuilt in memory on startup).

MEMORY:
  get_session_history()  → Session memory store. One ChatMessageHistory per session_id.

QUERY PROCESSING:
  rewrite_query()        → LLM-powered follow-up query rewriter. Preserves technical terms.
  expand_query()         → SAP alias resolution via regex + deduplication of repeated words.
  strip_stopwords()      → Remove filler words from BM25 query only. Vector gets full query.

RETRIEVAL:
  build_retriever()      → Hybrid two-stage retriever factory.
    hybrid_search()      → Merge BM25 + Vector by weighted score. Returns scored list.
    retrieve()           → Stage 1 threshold → Stage 2 fallback → circuit breaker.

RAG CHAIN:
  build_rag_chain()      → RAG chain with session memory via RunnableWithMessageHistory.
    format_docs()        → Join chunks with --- separators and [Source: ...] labels.
